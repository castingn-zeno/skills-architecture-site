Overview

왜 만들었나

한 명의 마케팅 리더가 "브랜드·디자인·콘텐츠·SEO·AEO·PR·고객·세일즈"를 모두 책임져야 하는 상황에서, Claude를 단순 어시스턴트가 아니라 "각 영역의 전담팀"처럼 작동시키기 위한 시스템입니다.

🧠
Before
하나의 만능 어시스턴트

매번 추측, 컨텍스트 누락, 톤 불일치, 표기 흔들림. 같은 정보를 매 대화마다 다시 설명해야 함. 산출물 일관성이 사람 의지에 의존.

⚡
After
18개 팀 스킬 + 메모리 + 1차자료

키워드 한 마디로 적합한 팀이 자동 호출, 메모리에서 컨텍스트 즉시 로드, 가드레일 자동 검수, 핸드오프 사양까지 표준화. 사람은 전략 결정과 최종 승인만.

Execution Process

요청에서 산출물까지 — 7단계

사용자 요청 한 줄이 어떻게 풀스택 산출물로 변환되는지의 전체 경로.

1

🎯 트리거 매칭 + 진입 스킬 결정

사용자 메시지의 키워드를 각 스킬의 description과 매칭해서 가장 적합한 진입 스킬 1개를 자동 선정합니다.

예) "랜딩페이지 새로 만들어줘"
→ 키워드 매칭: "랜딩페이지" → content-team 선정
→ Skill 도구로 호출
2

📚 컨텍스트 로드 (메모리 + 1차 자료)

MEMORY.md 인덱스 기반으로 25개 메모리 파일과 22건 회의록 중 작업에 필요한 것만 선별 로드합니다.

MEMORY.md (인덱스) ─┐
25개 메모리 파일 ├─→ 필요한 컨텍스트만 선택
22건 회의록 로데이터 ─┘
3

🤝 협업 라우팅 결정

진입 스킬은 작업 시작 시 협업 라우팅을 1줄로 명시합니다. 빠진 협업 = 가짜 산출물.

"이번 작업 라우팅:
  brand-team(가드레일) + customer-team(고객언어) +
  seo-team(메타) + aeo-team(FAQ구조) +
  design-team(시각사양) + dev-frontend(인터랙션)"
4

⚙ 작업 모드 선택 + 실행

각 스킬마다 6~7개 표준 작업 모드가 있습니다. 요청 유형에 맞는 모드를 선택해 실행합니다.

예) content-team Mode 5 (Landing Page)
[자료소집] → [구조잡기] → [본문작성] → [자가검수]
5

✅ 가드레일 자가 검수

brand-team이 정의한 가드레일을 모든 출력이 통과해야 합니다. 표기·금지어·보일러플레이트·UX라이팅·출처 검증.

□ ZENO/제노/캐스팅엔 표기 정확?
□ CFO 금지어 ("감으로 산다") 사용 여부?
□ 보일러플레이트 순서 (제노 앞 / 업무마켓9 뒤)?
□ 토스 UX 라이팅 8원칙 / 출처 주석 / 글자수
6

📦 산출물 + 핸드오프 사양

완성된 산출물에 더해, 다음 팀(디자인·개발·PR)이 받아 바로 작업할 수 있는 사양서를 함께 출력합니다.

[본문] + [근거 주석] + [핸드오프 사양]
→ dev-frontend: 인터랙션 사양표
→ design-team: 시각 토큰 사양
→ seo/aeo: 메타·JSON-LD
7

💾 영속화 (메모리 + 데일리 로그)

산출물에서 발견한 새 인사이트는 메모리에 갱신, 작업 자체는 데일리 로그에 Why/What/How 형식으로 기록.

데일리 로그 → /daily-logs/YYYY-MM-DD.md
메모리 갱신 → MEMORY.md + 25개 파일
"오늘 정리" 명령 시 → Notion DB로 푸시

Core Principles

모든 스킬 공통 원칙 7가지

어떤 팀이 호출되든 항상 지키는 약속.

01
🧠 메모리 우선
추측 금지. 작업 시작 전 메모리·1차 자료 먼저 확인.
02
🤝 협업 라우팅
혼자 작업 금지. 작업 시작 시 협업 팀을 1줄로 명시.
03
📎 출처 주석
모든 인용·통계는 출처 표기. 회의록·매체·페이지 명시.
04
🛡 가드레일 절대 양보 X
brand-team의 표기·금지어·슬로건은 어떤 상황에서도 변경 불가.
05
📦 핸드오프
다음 팀이 받아 바로 작업할 수 있는 사양 형태로 출력.
06
💾 영속화
데일리 로그 + 메모리 갱신. Why/What/How 3섹션 의무.
07
🇰🇷 한글 출력
모든 산출물은 한글. 기술 용어는 병기 가능.
